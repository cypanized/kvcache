01 · The crunch

AI workloads stress storage in three different ways

Each phase of the AI lifecycle wants something different from your storage. Build for one, and the others suffer.

TRAINING

Massive parallel reads

  • TBs of dataset shuffled across thousands of GPUs every epoch
  • Checkpoint storms — every node writes a fragment of model state at once
  • One slow read stalls the whole step
INFERENCE

Latency-critical reads

  • Cold start: 100s of GB of weights into HBM in seconds
  • KV cache offload — sub-ms reads of historical attention state
  • One stall and TTFT spikes for the user
RETRIEVAL · RAG

Many tiny random reads

  • Billions of small objects (vector embeddings, chunked docs)
  • Per query: dozens to thousands of point lookups
  • Tail latency, not throughput, is the SLO
Traditional storage forces a tradeoff — fast for one workload pattern, slow for the others. VAST runs all three on the same hardware, the same namespace.
02 · The architecture

DASE — Disaggregated Shared-Everything

Compute scales independently from storage. Every CNode sees every byte. The whole cluster looks like one disk.

The architecture

CNodes · stateless compute

  • All metadata + protocol logic; no state of their own
  • Add capacity by adding boxes — they all see the same pool
  • Failover is instant: another CNode takes over

DBoxes · shared NVMe + SCM

  • All-NVMe (QLC for capacity) + Storage Class Memory write buffer
  • Connected to every CNode over RDMA fabric — no shard ownership
  • One namespace, multi-protocol (NFS · SMB · S3 · NFSoRDMA)
Single namespace means: training data, checkpoints, KV cache, vector index, and inference artifacts all live in the same place — no copies, no rsync, no migration window when you move between phases.
03 · The lifecycle

Same data, six stages

From raw ingest to retired model — every stage hits storage differently. VAST exposes one namespace; the workload picks the protocol.

INGEST

Petabytes from data lakes, sensors, logs, vendor feeds. S3, NFS, parallel writes.

CURATE

Clean, dedupe, label. Data scientists hit the same files via NFS / S3 — no ETL.

TRAIN

Massive parallel reads to GPUs over NFSoRDMA / GPUDirect. Checkpoints land on the same pool.

TUNE

Fine-tune from a checkpoint that's already on the cluster. Versions sit side-by-side.

SERVE

Load weights into HBM. Offload KV cache. Serve a vector index. All from one namespace.

ARCHIVE

Demote to cold capacity (still in the same namespace). Retain for compliance.

04 · Training

Training: feeding GPUs and surviving checkpoint storms

Two opposite IO patterns hit the same storage at the same time — fast random reads from every GPU, and synchronized bulk writes from every node.

Read path · dataset shuffle

Every step, every GPU

  • Distributed shuffle: each GPU reads a different slice every epoch — no warm cache
  • NFSoRDMA / GPUDirect Storage drop CPU staging entirely
  • VAST aggregates TB/s read throughput across the GPU fleet
  • Slow tail-latency reads stall the whole iteration — NVMe + SCM keeps p99 tight

Write path · checkpoint storms

Every node, all at once

  • Distributed checkpoints: every rank writes a shard simultaneously
  • A 70B-class model: hundreds of GB written in one sync window
  • Concurrent ingest from thousands of nodes — no shard ownership to serialize on
  • Versioned snapshots make rollback and divergence-debugging trivial
Training spends GPU-seconds waiting for storage. Stalls compound across thousands of GPUs. VAST removes the tail because every CNode and every NVMe device is reachable from every GPU with predictable µs latency.
05 · Inference

Inference: the read path is everything

Three different reads share the same namespace — model weights, KV cache, and the vector store. Every one of them is on the critical path for time-to-first-token.

MODEL WEIGHTS

Cold start & hot reload

  • Hundreds of GB into HBM on cold start; seconds matter for autoscale
  • One artifact, one namespace — no per-node copies to invalidate
  • Hot-swap to a tuned variant without redistributing
KV CACHE

Offload & reuse

  • Long-context KV is too big to keep in HBM — offload to NVMe / object
  • Hash-keyed blocks = reuse across requests with shared prefixes
  • VAST is the storage substrate for LMCache + NIXL

VECTOR · RAG

Billions of small reads

  • Vector index sits on the same pool as everything else
  • Sub-ms point reads at billions-of-vector scale
  • Source documents stay alongside the embeddings — no separate object store
Every inference read is a tail-latency story. Whatever's slowest is your TTFT. VAST puts all three reads on the same low-latency, RDMA-attached pool — no protocol gymnastics, no inter-tier copies.
05 · Inference — vectors

Vector embeddings — meaning becomes geometry

A model maps each chunk of text (or image, or audio) to a high-dimensional vector. Items with similar meaning land in similar regions of the space. Retrieval is a nearest-neighbor search by cosine similarity.

2-dim toy space · clusters by meaning

Each word becomes a 2-dim point. The model learns to place similar meanings near each other — animals in one corner, vehicles in another, food in a third. In a real model the space is 768 or 1536 dimensions, but the geometry is the same.

3-dim space · cosine similarity finds neighbors

Cosine similarity = the cosine of the angle between two vectors. 0·9 + → very similar. ~0 → unrelated. negative → opposing meaning. Magnitude is ignored, so a long vector and a short vector pointing the same way are still similar.

Why this lives on storage — a production RAG index has billions of vectors. Each query computes similarities against tens of thousands of candidates. The whole pipeline is bottlenecked by random reads at sub-ms latency — exactly the workload VAST's all-NVMe + RDMA fabric is built for.
06 · Why VAST

Why VAST for AI

Six attributes that, together, make one storage platform viable across the whole AI lifecycle.

SINGLE NAMESPACE

No copies between phases

Training data, checkpoints, KV cache, vectors, archives — same paths, same permissions. No rsync, no S3-to-NFS migration window.

ALL-NVMe + SCM

Predictable µs latency

No spinning disks. QLC for capacity, Storage Class Memory as the always-fast write tier. p99 stays close to p50.

RDMA THROUGHOUT

Zero CPU on the data path

NFSoRDMA, GPUDirect Storage, S3-over-RDMA. Bytes go straight between NIC and GPU/NVMe — the host CPU is out of the way.

MULTI-PROTOCOL

NFS · SMB · S3 — same data

Data engineers use S3, ML pipelines use NFS, Windows analysts use SMB — all on the same files, no extra layer.

LINEAR SCALE

Exabyte capacity, TB/s throughput

Add CNodes for compute, DBoxes for capacity, independently. The architecture doesn't impose a "balance the racks" tax.

ONE PLATFORM

Whole AI lifecycle

Ingest, curate, train, tune, serve, archive — one fabric, one namespace, one operational model. No tier-of-tier-of-tier.
