LMCache & NIXL
Stop recomputing what you

Long-context inference is expensive because every request has to read the entire prompt before it can write a single token. LMCache reuses that work across requests. NIXL moves it across whatever storage and network you have.

01 · Phases of inference

Prefill vs. Decode — what each phase actually does

Every LLM request runs in two phases. They have different costs, different shapes, and different bottlenecks.

Phase 1 · Prefill

Read the prompt

The model ingests the entire input prompt and produces internal state for it.

Input
All N tokens of the prompt, at once
Work
One forward pass over all N tokens (highly parallel)
Output
The KV cache — per-layer key & value tensors for every token
Cost
Compute-bound · attention is ~O(N²)
Latency
This is what you wait for before the first token (TTFT)
Phase 2 · Decode

Write the answer

The model generates output tokens one at a time, each step reading the KV cache and appending to it.

Input
The KV cache from prefill + the previous token
Work
One small forward pass per output token (sequential)
Output
One token, then loop. KV cache grows by one entry per step.
Cost
Memory-bandwidth-bound · per-token cost is small
Latency
Determines tokens/second after the first token

03 · Inside the engine

From slab to pages — what paged attention actually does

One request's KV cache, before and after. Same data, same number of bytes — but the new organization is what every layer above (LMCache, NIXL, prefix sharing) is built on.

One request · 16 tokens of KV cache

What the engine actually changes

  • Same KV tensors, same total bytes — only the organization changes.
  • KV is laid out in a pool of fixed-size blocks (vLLM default: 16 tokens per block; we draw 4 here for clarity).
  • Each request keeps a block table: an array of physical block IDs, indexed by logical position.

What that unlocks

  • Hash & index any block by the prefix that produced it → LMCache lookups
  • Move any block to GPU / CPU / disk / network independently → NIXL transfers
  • Share a block by reference between requests → prefix sharing
  • Evict at block granularity instead of dropping a whole slab
03 · Inside the engine

Paged attention — KV memory in pages, not slabs

Before LMCache or NIXL come into play, the inference engine itself splits KV memory into fixed-size blocks. That partitioning is what makes blocks shareable, hashable, and addressable enough for an external cache to exist at all.

Why pages, not slabs

  • Pre-paged engines reserved a contiguous slab per request, sized to max sequence length. Most of it sat empty.
  • vLLM's paged attention treats KV memory like virtual memory: fixed-size blocks (typically 16 tokens), allocated on demand.
  • A per-request block table maps logical token positions → physical block IDs — same indirection as a CPU page table.
  • Multiple requests can point at the same physical block. Prefix sharing happens inside one engine.

Why this matters here

  • LMCache hashes blocks by the prefix that produced them — only meaningful because blocks are fixed-size, addressable, and content-stable.
  • NIXL transfers blocks between tiers — only tractable because a block is a uniform-sized unit, not a per-request slab.
  • Paged attention is the substrate. LMCache and NIXL extend it across requests, machines, and storage.

04 · The two layers

Together: who decides, who moves

LMCache and NIXL solve two halves of the same problem. LMCache decides what to reuse; NIXL gets it where it needs to go.

The stack

LMCache · the brain

  • Decides what KV cache is worth keeping
  • Indexes blocks by token-prefix hash
  • Looks up hits for incoming requests
  • Owns eviction, policy, multi-tenant keying

NIXL · the muscle

  • Moves KV between GPU, CPU, disk, network, object store
  • Abstracts a single buffer-list descriptor over all of them
  • Optimizes for zero-copy and async transfers
  • Plugs in backends: UCX, CUDA IPC, Mooncake, POSIX, S3
Crucially: NIXL is below LMCache, not above it. LMCache asks what exists and where; NIXL is the data path that gets it into GPU memory.

05 · LMCache

LMCache, in detail

A cache layer at the inference engine that treats KV cache as a first-class, addressable, reusable artifact.

What it is

  • Sits next to the inference engine (e.g. vLLM)
  • Maintains an index of KV cache blocks, keyed by hash of the token prefix that produced them
  • On each prompt, finds the longest matching prefix and asks NIXL to materialize those blocks into GPU memory
  • If a request misses, the engine prefills normally — and LMCache captures the resulting KV blocks for next time

What it does

  • Eliminates redundant prefill across requests with shared prefixes
  • Drops time-to-first-token dramatically on long, repeated contexts
  • Enables prefill / decode disaggregation: one machine prefills, another decodes — using the cache as the handoff
  • Supports multi-tenant policies through the lookup key (include tenant ID for isolation, omit it to share)

Tiering — where blocks actually live

Insertion

Newly prefilled blocks land in the fastest tier with capacity.

Demotion · LRU

When a tier fills, least-recently-used blocks spill down via NIXL — not dropped.

Promotion · on hit

Cold-tier hits get pulled back into GPU memory; warm-staying is configurable.

Placement is intentionally simple — recency + capacity. The smart parts are the prefix-hash lookup and NIXL's cross-tier movement. Operators tune capacity per tier, write-through behavior, chunk size, and tenant-key composition.

06 · NIXL

NIXL — one buffer list, many backends

A unified data-movement layer for KV cache. Below LMCache, above the messy reality of GPU memory, host RAM, NVMe, networks, and object stores.

The problem it solves

KV cache lives somewhere — and where varies. A hot block sits in GPU HBM. A warm block sits in host CPU memory. A cold block sits on local NVMe. A cross-node block lives on another machine. A historical block lives in object storage. Each has its own API and transfer model.

Without an abstraction, every higher-level system has to write code for every backend it wants to support. NIXL gives them one.

The core abstraction

nixl_xfer_desc {
  entries: [
    { dev: gpu0,   addr: 0x7f...,  len: 8 KB },
    { dev: cpu,    addr: 0x55...,  len: 8 KB },
    { dev: nvme0,  offset: 0x4000, len: 32KB },
    { dev: s3,     key: "kv/abcd", len: 64KB },
    { dev: node3,  addr: 0x9c...,  len: 8 KB },
  ],
  async: true,
  zero_copy: true
}

06 · NIXL · why it's fast

Two design wins: zero-copy and async

Beyond unifying the API, NIXL also moves bytes faster than naive paths would — by skipping CPU staging and by never blocking the GPU.

Win 1

Zero-copy, direct paths

NIXL moves data directly between endpoints — but only when the backend speaks RDMA (GPU↔NIC, GPU↔NVMe via GPUDirect, S3-over-RDMA). Plain HTTP object storage still stages through a CPU buffer.

Win 2

Async / non-blocking

Transfers return immediately. The GPU keeps computing while data moves in the background.
